# 面试点

### 关于向量数据库和Elasticsearch

* 你提到用 **Elasticsearch** 同时作为文档数据库和向量数据库。为什么选择它而不是更专业的向量数据库，比如 Milvus、Weaviate 或 Faiss？你认为 Elasticsearch 在作为向量数据库时有什么优势和劣势？

> 我们的系统本身就依赖强大的全文检索能力。Elasticsearch完美地在一个引擎内同时提供了全文检索和向量检索，避免了维护两套不同数据库（如ES+Milvus）带来的数据同步、运维和系统复杂性成本。 ES允许我们执行复杂的混合查询。例如，可以非常方便地先通过元数据（如doc_id, author）过滤文档，再在过滤后的文档子集上进行向量相似度搜索，或者将关键词得分与向量得分进行线性加权融合。这在纯向量数据库中实现起来更为复杂。

* 在 Elasticsearch 中，你是如何存储向量的？具体用的是什么字段类型？（例如 `dense_vector`）

> 使用Elasticsearch官方推荐的 dense_vector 字段类型来存储文档块经过Embedding模型（如bge-large-zh）计算出的浮点数向量。

* Elasticsearch 在进行向量检索时，底层用的什么算法？你是如何配置这个算法的，比如选择 `HNSW`（Hierarchical Navigable Small World）还是其他方法？

> HNSW是一种基于图论的算法，通过构建一个分层的、可导航的小世界图，使得搜索过程可以从粗到精，快速逼近目标向量的最近邻，避免了暴力计算所有距离的开销。

* 在生产环境中，当索引的文档数量达到千万甚至亿级别时，你认为 Elasticsearch 的向量检索性能会面临什么挑战？你会如何进行优化？（例如分片、副本、索引优化等）

>  HNSW索引完全驻留在内存中以保证性能。亿级高维向量会消耗大量堆内存，可能引发GC问题甚至OOM。根据数据量和集群节点数，进行合理的分片。将一个大索引分散到多个节点上，利用分布式并行搜索提高吞吐量。通常一个分片大小建议在20-50GB。

### 关于检索策略和多路召回

* 你提到了**全文检索**和**语义检索**。请具体解释一下这两种检索方式在你系统中的实现逻辑。

> 全文检索直接利用Elasticsearch强大的倒排索引。对用户查询进行分词后，使用match或match_phrase等查询，在文档的content字段进行搜索，按ES默认的BM25相关性评分进行排序。语义检索利用Embedding模型（如bge-large-zh）计算文档块的向量表示，并使用HNSW索引进行快速搜索。

* 你是如何实现**多路召回**的？具体是哪几路？除了全文和语义，是否还考虑了其他召回方式？（例如关键词匹配、基于元数据的过滤等）

> 使用knn搜索（HNSW最主流），召回与查询语义最相关的Top K个文档块。使用BM25搜索，召回与查询关键词匹配度最高的Top K个文档块。

* **多路召回合并**的逻辑是怎样的？你是简单地将不同召回结果合并在一起，还是根据某些规则或权重进行融合？

> 通过RRF合并

* **重排序**（Reranking）是RAG系统性能的关键。你是如何实现重排序的？用了哪些模型或算法？（例如使用 cross-encoder 模型）

> 将多路召回合并后的Top N个候选文档块（例如，N=50）和原始用户查询，一起输入一个更精细但更耗时的重排模型。我们使用的是Cross-Encoder模型（如bge-reranker-large）。

* 为什么需要重排序？重排序前后，检索结果的质量和排序有什么具体的变化？你是如何评估这个变化的？

>  排序主要基于浅层的语义和关键词匹配，可能会把“看似相关但实际答非所问”的文档排在前列。 Cross-Encoder能更准确地判断文档是否真正回答了问题。例如，它能把一个包含精确答案但用词与查询差异较大的文档排到顶部，而将一个频繁出现关键词但逻辑不通的文档排到后面。

### 关于知识库管理和系统架构

* **知识库管理**是一个很实际的问题。你的系统是如何实现知识库的更新、增删和维护的？当知识库内容发生变化时，如何保证向量索引的同步更新？

> 提供一个管理API，供用户上传、更新或删除文档。上传的文档（PDF, Word等）被送入解析服务，进行文本提取、清洗和分块。

* **支持多人同时问答**，这意味着你的系统需要考虑并发和性能。你是如何设计系统架构来处理高并发请求的？（例如，使用消息队列、负载均衡、服务拆分等）

> RAG服务本身是无状态的，所有状态（文档、向量、会话）都存储在后端数据库（ES）和缓存中，方便水平扩展。 Elasticsearch本身就是一个分布式集群，可以通过增加数据节点来提升检索的并发处理能力。 

* 整个系统的技术栈是什么样的？（例如，前端、后端服务、数据处理管道等）

> 后端API： Python (FastAPI / Flask); 向量/文档数据库： Elasticsearch
